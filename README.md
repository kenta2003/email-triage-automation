# email-triage-automation — 複数 Gmail アカウントの AI トリアージ & Slack 通知システム

複数の Gmail アカウントに届くメールを自動取得し、LLM で内容を分類・要約・優先度判定したうえで、Slack の通知チャンネルを優先度別に振り分けて通知するパーソナル自動化システム。重要度の高いメールと確認優先度の低いメールでチャンネルを分けることで、すべてのメールを通知しつつ、対応すべきメールを優先的に把握できる構成にした。

**概要**
- 個人開発で要件定義 → 設計 → 実装 → 定期実行までを一気通貫で構築した、Python + Gmail API + Claude + Slack ベースのメール トリアージ システム
- LLM による分類とルールベース分類を組み合わせた二段構えで、AI 判定の柔軟性とルール判定の安全性を両立
- Claude Code CLI（`claude -p`）をサブプロセス呼び出しすることで、追加の従量課金 API を使わずに分類・要約処理を実装
- Kiro 方式の Spec-Driven Development（要件 → 設計 → タスク → 実装）に沿って仕様を `.kiro/specs/` 配下に整備

> 本プロジェクトは個人開発のポートフォリオです。OAuth クレデンシャル等を設定すればローカル環境で動作確認が可能です。

## 1. 作成物の説明（目的・背景）

### 目的・背景
日常的に複数の Gmail アカウントを確認する中で、重要度の低いメールに時間を取られ、本当に確認すべき連絡の把握が遅れることに課題を感じた。特に、メールは件名や送信者だけでは重要度を判断しにくく、本文の文脈を踏まえて対応要否を判断する必要がある。そこで、メール取得、内容理解、分類、要約、通知までを自動化し、確認すべきメールの優先度を整理できる仕組みを個人で開発した。メール確認にかかる時間を減らし、重要な連絡の見落としや返信遅れを防ぐことを目的としている。

### 主な機能
- **マルチアカウントの自動取得**: 複数の Gmail アカウントから未処理メールの件名・送信者・本文・受信日時を Gmail API で取得
- **AI 分類・要約**: Claude を用いて要約 / 対応要否 / 優先度 / 分類ラベルを JSON で生成
- **ルールベース併用判定**: 送信元・件名・本文キーワードによるルールで AI 分類の補正・フォールバックを実施
- **Slack 通知（優先度別チャンネル振り分け）**: 即時返信用 / 重要確認用 / 一般確認用 / メルマガ用の 4 チャンネルへ自動振り分け
- **返信下書き生成**: 返信が必要と判定されたメールには返信下書きを生成し、Slack 通知本文に添付
- **重複通知防止**: 処理済みメールに専用 Gmail ラベルを付与し、次回実行時の再通知を防止
- **アカウント単位の Graceful Degradation**: 片方のアカウントで認証 / API エラーが発生しても、他方の処理を継続
- **定期実行**: Windows Task Scheduler から `run.py` を一定間隔で起動し無人運用

## 2. 自身が担当した役割

個人開発のため、要件定義から運用設計まで全工程を担当。

### 設計
- 要件定義（EARS フォーマットでの要件記述・受け入れ条件の整理）
- アーキテクチャ設計（`sources` / `processor` / `notifiers` の 3 層分離、`BaseSource` / `BaseNotifier` 抽象基底クラスによる拡張性確保）
- 分類カテゴリ設計（4 段階の優先度と対応する Slack チャンネル設計）
- プロンプト設計（JSON 出力の構造定義、バッチサイズ、リトライ戦略）
- Kiro 方式の Spec-Driven Development による仕様整備（`.kiro/specs/email-triage-automation/`）

### 実装
- **メール取得**: Gmail API v1 クライアント、OAuth 2.0 認証フロー、未処理メールのクエリ抽出、処理済みラベル付与
- **AI 分類・要約**: Claude Code CLI（`claude -p`）のサブプロセス呼び出し、最大 50 件のバッチ分類、JSON パース、最大 3 回リトライ
- **ルールベース分類**: 送信元ドメイン / 件名 / 本文キーワードによる判定、AI 結果との合成（昇格ロジック、フォールバック）
- **Slack 通知**: Incoming Webhook クライアント、4 チャンネル振り分け、ダイジェスト整形、返信下書き付き個別通知
- **オーケストレーター**: `run.py` によるアカウント単位の Graceful Degradation、構造化ログ
- **設定管理**: `python-dotenv` + `dataclass` による型付き不変設定

### 運用設計
- Windows Task Scheduler による定期実行（`run.py` を一定間隔で起動）
- 構造化ログによる実行結果の可観測性確保（`logs/` 配下に出力）
- OAuth クレデンシャル / トークン / `.env` を `.gitignore` で完全除外する運用ルール
- 返信は**下書き生成のみ**（自動送信しない）とする安全側の運用方針

## 3. 直面した課題と解決方法

### 課題1: 件名や送信者だけでは対応要否を判断できない
**課題**: メールは件名や送信者だけでは対応要否を判断しにくく、本文の文脈を踏まえて優先度を判断する必要があった。単純なキーワード判定では、文面の含意やトーンに左右される判断ができない。

**解決アプローチ**:
- 件名・送信者・本文の一部をプロンプトに渡し、Claude に要約 / 対応要否 / 優先度 / 分類ラベルを JSON 形式で出力させる構成にした
- バッチ単位で複数件を一度に分類することで、CLI 起動コストを抑えつつ文脈ベースの判断を実用速度で実行できるようにした
- これにより、件名だけでは判断できないメール（例: 一見メルマガに見えて中に締切連絡が含まれているもの等）にも内容に応じた分類ができるようにした

### 課題2: 通知量が多いと重要メールが埋もれる
**課題**: すべてのメールを同じ場所に通知すると、重要度の低い通知に埋もれて本当に対応すべきメールを見落とす可能性があった。一方で、低優先度メールを通知しないと、まれに含まれる重要連絡が把握できなくなる。

**解決アプローチ**:
- LLM による分類結果をもとに、Slack の通知チャンネルを **4 つに分割**（即時返信が必要なメール / 重要な確認事項 / 一般的な確認事項 / メルマガ・自動通知）
- 即時返信が必要なメールは **1 件ずつ個別通知**（返信下書き付き）、それ以外は**ダイジェスト形式**で通知することで、通知量を維持しながら確認優先度を整理
- 通知内容には送信者・件名・要約・優先度・対応要否を含め、Gmail を開かなくても Slack 上で概要を把握できる構成にした

### 課題3: LLM の分類結果をそのまま信頼すると誤判定リスクがある
**課題**: LLM による分類は文脈を踏まえられる一方で、判断が常に一定とは限らないため、分類結果をそのまま信頼しすぎると通知先の誤りが起こる可能性があった。明らかに重要なメール（特定取引先・期限通知・本人確認等）が低優先度に分類されると、見落としに直結する。

**解決アプローチ**:
- AI 分類に加えて、**送信元ドメイン・件名・本文キーワードに基づくルールベース判定**を組み合わせた
- ルールで「重要」と判定されたメールは、AI が低優先度と判定していても上位優先度に**強制昇格**させるロジックを実装
- JSON パース失敗時は最大 **3 回までリトライ**し、それでも失敗した場合はルールベース分類に**フォールバック**することで、AI 障害時にも通知が止まらない設計にした
- AI による柔軟な判断と、ルールによる安全側の判定を組み合わせることで、実用上の安定性を確保した

### 課題4: 複数アカウント運用時の片側障害を波及させない
**課題**: 複数 Gmail アカウントを 1 プロセスで扱う構成では、片方のアカウントで OAuth トークン失効や API エラーが発生したときに、もう片方のアカウントの処理まで止まると運用上のリスクが大きい。

**解決アプローチ**:
- アカウント単位で処理ループを `try/except` で隔離し、片方のアカウントで例外が発生しても他方の処理を継続する **Graceful Degradation** 構成にした
- アカウントごとに専用の処理済みラベル（`Claude-Processed-*`）を発行し、Gmail 側で永続的に処理状況を管理することで、次回実行時に同じメールが再通知されないようにした
- ログには各アカウントの取得件数 / 分類成否 / 通知結果を構造化して記録し、後追いで原因特定できるようにした

## 4. 技術情報

### アーキテクチャ概要

```
[ Windows Task Scheduler ] (定期実行)
        │
        ▼
[ run.py — Orchestrator ]
   ├── アカウント単位の try/except による Graceful Degradation
   ├── 構造化ログ (logs/)
   │
   ├── sources/gmail.py
   │     ├── OAuth 2.0 認証
   │     ├── 未処理メール取得 (Gmail API v1)
   │     └── Claude-Processed-* ラベルによる重複排除
   │
   ├── processor.py
   │     ├── 最大 50 件のバッチ プロンプト構築
   │     ├── claude -p CLI サブプロセス呼び出し (Claude)
   │     ├── JSON パース + 最大 3 回リトライ
   │     ├── ルールベース分類 (送信元 / 件名 / 本文キーワード)
   │     └── AI 結果との合成 (昇格ロジック + フォールバック)
   │
   └── notifiers/slack.py
         ├── Slack Incoming Webhook (4 チャンネル)
         ├── 即時返信用: 1 件ずつ個別通知 + 返信下書き
         └── その他: ダイジェスト整形
```

### 技術スタック

| レイヤ | 技術 | 選定理由 |
|---|---|---|
| 言語 / ランタイム | Python 3.10+ | Gmail API SDK の充実、Windows との親和性、`dataclass` による型付き設定 |
| LLM | Anthropic Claude（Claude Code CLI `claude -p`） | サブスクリプション内で完結し、追加の従量課金 API が不要 |
| メール取得 | Gmail API v1 + OAuth 2.0（`google-api-python-client`, `google-auth-oauthlib`） | 公式 SDK、ラベル機能による重複排除を Gmail 側で完結できる |
| 通知 | Slack Incoming Webhook | OAuth フロー不要で軽量、チャンネル単位で URL を分けるだけで多チャンネル振り分けが可能 |
| スケジューラ | Windows Task Scheduler | 追加ランタイム不要、ローカル運用に十分 |
| 設定管理 | `python-dotenv` + `dataclass` | 型付きの不変設定でテスタビリティを確保 |
| ログ | Python `logging`（構造化フォーマット） | 標準ライブラリで完結、後追い調査用に十分 |
| 開発手法 | Kiro 方式 Spec-Driven Development | 要件 → 設計 → タスク → 実装の各段階を `.kiro/specs/` に明文化 |

### アーキテクチャ設計
- **3 層分離**: `sources`（メール取得）/ `processor`（AI 分類）/ `notifiers`（通知）を独立モジュールとして分離
- **抽象基底クラス**: `BaseSource` / `BaseNotifier` を定義し、将来別のメールソース（IMAP・arXiv 等）や別の通知先（Discord 等）を追加する際、既存コードを変更せずに拡張可能
- **オーケストレーション**: `run.py` はソース・プロセッサ・ノーティファイアを呼び出すだけの薄い層に保ち、テスト容易性を確保

### メール取得（Gmail API + OAuth 2.0）
- OAuth 2.0 認証で Gmail アカウントへのアクセス権限を取得し、件名 / 送信者 / 本文 / 受信日時を取得
- 取得対象は未処理メールに限定（Gmail クエリで `-label:Claude-Processed-*` を指定）
- 処理完了後にアカウントごとの専用ラベルを付与し、次回実行時に同じメールを再取得しない構成
- アカウントごとに独立した OAuth クレデンシャル / トークンを扱い、`tokens/` 配下にアカウント別に保存

### AI 分類・要約（Claude）
- Claude Code CLI（`claude -p`）をサブプロセス起動し、プロンプトを標準入力経由で渡して JSON 出力を受け取る方式
- 1 回のプロンプトに最大 **50 件** のメールをまとめて渡すバッチ構成（CLI 起動コストの抑制）
- 出力 JSON のスキーマは要約 / 対応要否 / 優先度 / 分類カテゴリ / 返信下書き
- JSON パース失敗時は最大 **3 回** リトライし、それでも失敗した場合はルールベース分類にフォールバック

### ルールベース分類（AI との併用）
- 送信元ドメイン / 件名キーワード / 本文中の特定文字列に基づくルールを `config.py` で定義
- AI が低優先度と判定したメールでも、ルールで「重要」と判定されたものは上位優先度へ**強制昇格**
- AI 障害時のフォールバック先としても利用し、通知の継続性を担保

### Slack 通知（4 チャンネル振り分け）

| 優先度 | チャンネル用途 | 通知形式 |
|---|---|---|
| 即時返信必要 | 個別問い合わせ / 期限付き案件 | 1 件ずつ個別通知（返信下書き付き） |
| 重要 / 要確認 | 重要なお知らせ / FYI | ダイジェスト |
| 一般 / 要確認 | 一般的な確認事項 | ダイジェスト |
| スキップ | メルマガ / 自動通知 | 件名のみダイジェスト |

- 通知内容には送信者・件名・要約・優先度・対応要否を含め、Gmail を開かなくても概要を把握できる構成
- Slack Incoming Webhook を用いることで OAuth フローを避けて軽量に実装

### 例外処理・運用設計
- **アカウント単位の Graceful Degradation**: アカウントごとに `try/except` で囲み、片方の失敗を他方に波及させない
- **JSON パースの 3 回リトライ + ルールベース フォールバック**: AI 出力の不安定性に対する多段防御
- **構造化ログ**: 各アカウントの取得件数 / 分類成否 / 通知結果を `logs/` に出力し、後追い調査を容易化
- **重複通知防止**: Gmail ラベルで処理済み状態を Gmail 側に永続化（プロセス側に状態を持たない）

### セキュリティ設計
- OAuth クレデンシャル / トークン / `.env` を `.gitignore` で完全除外
- Slack Webhook URL は環境変数（`.env`）経由のみで管理
- 返信は**下書き生成のみ**（自動送信は行わない）
- LLM に渡すのは件名・送信者・本文の一部のみで、添付ファイル等は対象外

### 仕様駆動開発
- Kiro 方式の Spec-Driven Development（要件 → 設計 → タスク → 実装）で進行
- 仕様書は [`.kiro/specs/email-triage-automation/`](./.kiro/specs/email-triage-automation/) に格納
  - `requirements.md`: EARS フォーマットでの要件定義
  - `design.md`: 技術設計（アーキテクチャ・データフロー・例外処理方針）
  - `tasks.md`: 実装タスク分解
  - `spec.json`: 仕様メタデータ

## 5. ローカル環境での起動

### 前提
- Python 3.10 以上
- [Claude Code CLI](https://docs.claude.com/claude-code) インストール済み
- Gmail（OAuth 認証可能なアカウント）
- Slack ワークスペース（Incoming Webhook を作成可能）

### 手順
```bash
# 依存関係のインストール
pip install -r requirements.txt

# 環境変数ファイルの作成
cp .env.example .env
# .env に Slack Webhook URL を 4 本記入

# Google Cloud OAuth クライアント ID を作成し、
# credentials/primary_credentials.json として配置
# （複数アカウントなら secondary_credentials.json も）

# 初回認証（ブラウザが開きトークンが tokens/ に保存される）
python run.py

# Windows Task Scheduler に定期実行を登録
#   プログラム: python
#   引数:       run.py
#   開始場所:   <このフォルダの絶対パス>
```
