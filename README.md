# email-triage-automation

複数の Gmail アカウントに届くメールを自動取得し、LLM で分類・要約・優先度判定を行ったうえで、Slack の通知チャンネルを優先度別に振り分けて通知するパーソナル自動化システム。

> Windows Task Scheduler で定期実行し、すべてのメールを通知しつつ、対応すべきメールを優先的に把握できるようにした個人開発プロジェクト。

---

## 作成物の説明

複数の Gmail アカウントに届くメールを自動取得し、LLM を用いて内容の分類、要約、優先度判定を行った上で、Slack に通知するシステムを個人で開発した。メール確認にかかる時間を減らし、重要な連絡の見落としや返信遅れを防ぐことを目的としたものである。重要度の高いメールと確認優先度の低いメールで Slack の通知チャンネルを分けることで、すべてのメールを通知しつつ、対応すべきメールを優先的に把握できるようにした。

## 目的・背景

日常的に複数の Gmail アカウントを確認する中で、重要度の低いメールに時間を取られ、本当に確認すべき連絡の把握が遅れることに課題を感じた。特に、メールは件名や送信者だけでは重要度を判断しにくく、本文の文脈を踏まえて対応要否を判断する必要がある。そこで、メール取得、内容理解、分類、要約、通知までを自動化し、確認すべきメールの優先度を整理できる仕組みを作成した。

## 自身が担当した役割

個人開発のため、以下の全工程を一人で担当した。

- 要件定義（EARS フォーマットでの要件記述・受け入れ条件の整理）
- アーキテクチャ設計（ソース層・処理層・通知層の分離、抽象基底クラス設計）
- Gmail API 連携 / OAuth 2.0 認証
- メール取得処理（未処理メールの抽出、ラベルによる重複排除）
- LLM による分類・要約処理（プロンプト設計、JSON 出力の安定化、リトライ制御）
- ルールベース分類との併用ロジック
- Slack 通知（優先度別チャンネル振り分け、ダイジェスト整形）
- 重複通知防止（Gmail ラベルによる処理済み管理）
- 例外処理（アカウント単位の Graceful Degradation）
- 定期実行設定（Windows Task Scheduler）
- 仕様書整備（Kiro 方式の Spec-Driven Development、`.kiro/specs/` 配下に格納）

## 直面した課題と解決方法

### 課題 1: 件名や送信者だけでは重要度を判断できない

メールは件名や送信者だけでは対応要否を判断しにくく、本文の文脈を踏まえて優先度を判断する必要があった。そこで、件名、送信者、本文の一部を LLM に渡し、要約、対応要否、優先度、分類ラベルを出力する構成にした。これにより、単純なキーワード判定では判断しにくいメールも、内容に応じて分類できるようにした。

### 課題 2: 通知量が多いと重要メールが埋もれる

すべてのメールを同じ場所に通知すると、重要度の低い通知に埋もれて本当に対応すべきメールを見落とす可能性があった。そこで、LLM による分類結果をもとに、重要度の高いメールと確認優先度の低いメールで Slack の通知チャンネルを分ける設計にした。重要なメールはすぐ確認できるチャンネルへ、緊急度の低いメールは別チャンネルへ通知することで、通知量を維持しながら確認優先度を整理した。

### 課題 3: LLM の分類結果をそのまま信頼すると誤判定リスクがある

LLM による分類は文脈を踏まえられる一方で、判断が常に一定とは限らないため、分類結果をそのまま信頼しすぎると通知先の誤りが起こる可能性があった。そこで、AI 分類に加えて、送信元やキーワードに基づくルールベースの判定も組み合わせ、明らかに重要なメールが低優先度に分類されないようにした。AI による柔軟な判断と、ルールによる安全側の判定を組み合わせることで、実用上の安定性を高めた。

加えて、JSON 出力のパース失敗に備えて最大 3 回のリトライを実装し、リトライ後も失敗した場合はルールベース分類にフォールバックする多段防御構成とした。

## 技術情報

### 使用モデル・API

| 用途 | 採用技術 |
|------|---------|
| LLM（分類・要約・優先度判定） | Anthropic Claude（Claude Code CLI `claude -p` をサブプロセス呼び出し） |
| メール取得 | Gmail API v1 + OAuth 2.0 |
| 通知 | Slack Incoming Webhook（4 チャンネル構成） |
| 定期実行 | Windows Task Scheduler |
| 言語・ランタイム | Python 3.10+ |
| 設定管理 | `python-dotenv` + `dataclass`（型付き不変設定） |

LLM 呼び出しは Claude Code CLI の `claude -p` をサブプロセス起動する形で実装し、追加の従量課金 API を使わずに分類・要約処理を完結させた。

### アーキテクチャ

```
┌─────────────────────────────────────────────────────────────┐
│  Windows Task Scheduler（定期実行）                          │
└─────────────────────────────┬───────────────────────────────┘
                              ▼
        ┌────────────────────────────────────────┐
        │           run.py（オーケストレーター）   │
        │   - アカウント単位の Graceful Degradation │
        │   - 構造化ログ                          │
        └──────┬──────────────┬──────────────────┘
               │              │
   ┌───────────▼──┐    ┌──────▼───────────┐
   │ sources/     │    │ processor.py     │
   │  gmail.py    │───▶│  - 最大 50 件     │
   │  - OAuth 2.0 │    │    のバッチ分類   │
   │  - ラベルで   │    │  - claude -p CLI │
   │    重複排除   │    │  - 3 回リトライ + │
   └──────────────┘    │    ルール分類    │
                       │    フォールバック  │
                       └──────┬───────────┘
                              ▼
                       ┌──────────────────┐
                       │ notifiers/       │
                       │  slack.py        │
                       │  - 優先度別      │
                       │    4 チャンネル   │
                       │  - 返信下書き付与 │
                       └──────────────────┘
```

層を `sources` / `processor` / `notifiers` の 3 層に分離し、それぞれ `BaseSource` / `BaseNotifier` の抽象基底クラスを介してやり取りする構成とした。将来別のメールソース（IMAP・arXiv 等）や別の通知先（Discord 等）を追加する際、既存コードを変更せずに拡張できる。

### 実装方法

#### 1. メール取得（Gmail API + OAuth 2.0）

- OAuth 2.0 認証で Gmail アカウントへのアクセス権限を取得し、件名・送信者・本文・受信日時を取得
- 取得対象は未処理メールに限定（Gmail のクエリで `-label:Claude-Processed-*` を指定）
- 処理完了後にアカウントごとの専用ラベルを付与し、次回実行時に同じメールを再取得しないようにする
- 複数アカウントはアカウント単位で処理ループを分離し、片方のアカウントで認証エラーや API エラーが発生しても、他方のアカウントの処理を継続できるようにした

#### 2. AI 分類・要約（Claude）

- 取得したメール情報（件名・送信者・本文の冒頭）をプロンプトに渡し、以下を JSON で出力させる
  - 要約（数行）
  - 対応要否（要返信 / 要確認 / スキップ）
  - 優先度ラベル
  - 分類カテゴリ
  - 返信が必要なメールには返信下書きも生成
- 出力 JSON のパース失敗時は最大 3 回までリトライし、それでも失敗した場合はルールベース分類にフォールバック
- 1 回の呼び出しで最大 50 件をバッチ処理することで、CLI 起動コストを抑制

#### 3. ルールベース分類との併用

- 送信元ドメイン・件名キーワード・本文中の特定文字列に基づくルールを定義
- AI 分類が「低優先度」と判定したメールでも、ルールで「重要」と判定されたものは強制的に上位優先度へ昇格
- AI の柔軟性とルールの確実性を組み合わせ、誤判定によるリスクを低減

#### 4. Slack 通知（優先度別チャンネル分離）

- AI が判定した優先度・対応要否に応じて、4 つの Slack チャンネルへ振り分け
  - 即時返信が必要なメール: 1 件ずつ個別通知（返信下書き付き）
  - 重要な確認事項: ダイジェスト通知
  - 一般的な確認事項: ダイジェスト通知
  - メルマガ・自動通知等: 件名のみのダイジェスト
- 通知内容には送信者・件名・要約・優先度・対応要否を含め、Gmail を開かなくても概要を把握できる構成
- Slack Incoming Webhook を用いることで、OAuth フローを避けて軽量に実装

#### 5. 定期実行・運用

- Windows Task Scheduler で一定間隔ごとに `run.py` を実行
- 1 回の実行で「メール取得 → AI 分類 → Slack 通知 → 処理済みラベル付与」までを自動完了
- 構造化ログを `logs/` に出力し、後から個別実行の挙動を追跡できる
- OAuth クレデンシャル / トークン / `.env` は `.gitignore` で完全除外し、リポジトリには機微情報を含めない

### 仕様駆動開発

Kiro 方式の Spec-Driven Development（要件定義 → 設計 → タスク分解 → 実装）で進めた。仕様書は [`.kiro/specs/email-triage-automation/`](./.kiro/specs/email-triage-automation/) に格納している。

## セットアップ

### 前提
- Python 3.10+
- [Claude Code CLI](https://docs.claude.com/claude-code) インストール済み
- Gmail（OAuth 認証可能なアカウント）
- Slack ワークスペース（Incoming Webhook を作成可能）

### 手順

```bash
# 1. 依存関係インストール
pip install -r requirements.txt

# 2. 環境変数の設定
cp .env.example .env
# .env に Slack Webhook URL を 4 本記入

# 3. Google Cloud OAuth クライアント ID を作成し、
#    credentials/primary_credentials.json として配置
#    （複数アカウントなら secondary_credentials.json も）

# 4. 初回認証（ブラウザが開きトークンが tokens/ に保存される）
python run.py

# 5. Windows Task Scheduler に定期実行を登録
#    プログラム: python
#    引数:       run.py
#    開始場所:   <このフォルダの絶対パス>
```

## セキュリティ

- OAuth クレデンシャル / トークン / `.env` は `.gitignore` で完全除外
- Slack Webhook URL は環境変数経由のみ
- 返信は**下書き生成のみ**（自動送信しない）
